# PRD - LIFT 구현 기반 제품 요구사항

작성일: 2026-07-08
대상 프로젝트: LIFT
작성 기준: 현재 저장소의 백엔드, 프론트엔드, 로컬 동작 검증 결과를 기준으로 작성한다.

## 1. 문서 목적

이 문서는 LIFT 프로젝트의 실제 구현 기능을 제품 요구사항 형태로 정리한다. 심사자가 PRD와 코드를 비교하더라도 과장으로 보이지 않도록, 구현된 기능과 데모/모의 구현, 환경 설정이 필요한 기능, 추후 확장 가능 기능을 구분한다.

이 PRD는 다음 내용을 의도적으로 포함하지 않는다.

- 실제 행정기관 신청을 대행한다는 주장
- 실제 공공 마이데이터/전자문서지갑 연동 완료 주장
- AI가 법적 수급 자격을 확정한다는 주장
- 토스 실거래 결제가 운영 결제로 완성됐다는 주장
- 모든 생애 이벤트를 지원한다는 주장

## 2. 제품 개요

LIFT는 퇴직 또는 이직 상황에서 사용자가 확인해야 할 행정 절차, 예상 수령/절감 금액, 필요 서류, 공식 신청 링크, 추가 공공혜택 후보를 한 번에 정리해 주는 생애전환 행정 준비 서비스다.

MVP의 핵심 가치는 자동 신청이 아니라, 사용자의 입력 정보를 바탕으로 "지금 어떤 절차를 먼저 확인해야 하는지"를 설명 가능한 리포트로 제공하는 것이다.

## 3. 문제 정의

퇴직, 계약 만료, 이직 직후 사용자는 실업급여, 건강보험, 국민연금, 퇴직금, 세금 정산처럼 여러 제도를 동시에 확인해야 한다. 각 제도는 신청 조건, 기한, 준비 서류, 공식 신청 채널이 흩어져 있어 사용자가 우선순위를 판단하기 어렵다.

LIFT는 이 문제를 다음 방식으로 해결한다.

- 사용자의 퇴직/이직 상황을 짧은 진단 폼으로 수집한다.
- 서버 룰 엔진이 절차별 신청 가능성, 우선순위, 필요 서류를 계산한다.
- 결제 전에는 핵심 미리보기를 제공하고, 결제 후에는 전체 행정 로드맵을 제공한다.
- 공공혜택 캐시 DB에서 추가 혜택 후보를 조회하고 점수화한다.
- OpenAI가 활성화된 환경에서는 공공혜택 후보의 우선순위와 쉬운 요약을 보조 생성한다.

## 4. 대상 사용자

- 퇴직 예정자 또는 퇴직 직후 사용자
- 계약 만료, 폐업/도산, 해고 등으로 구직급여 가능성을 확인해야 하는 사용자
- 이직 중 건강보험/국민연금/세금 정산 공백을 확인해야 하는 사용자
- 신청 가능성이 있는 제도와 공식 신청 링크를 한 화면에서 보고 싶은 사용자
- 데모 환경에서 LIFT의 진단, 리포트, 결제, PDF, AI 질문 흐름을 확인하는 심사자

## 5. MVP 범위

### 5.1 포함 범위

- 카카오/네이버 소셜 로그인 시작 및 콜백 처리
- 데모용 브라우저 로컬 세션
- 약관 동의 및 사용자 프로필 일부 저장
- 퇴직/이직 생애 이벤트 선택
- 퇴직/이직 진단 입력
- 룰 엔진 기반 리포트 생성
- 결제 전 미리보기
- BASIC/PLUS 플랜 선택
- 데모 결제 및 토스 테스트 결제 승인 API
- 결제 완료 리포트 상세 조회
- 예상 수령액/절감액 계산
- 필요 서류 목록 및 모의 서류 조회
- PLUS 리포트 PDF 저장 화면
- PLUS 리포트 기반 AI 질문 10회
- 공공혜택 캐시 DB 기반 추가 혜택 추천
- OpenAI 활성화 시 공공혜택 AI 재랭킹/요약
- 퇴직/이직 커뮤니티 게시판

### 5.2 제외 범위

- 실제 행정기관 신청서 제출
- 실제 공공 마이데이터/전자문서지갑 서류 발급
- 운영 PG 정산 및 환불 관리
- 관리자용 공공혜택 수집/검수 화면
- 결혼, 출산휴가, 육아휴직 등 추가 생애 이벤트의 전체 리포트
- 법적 자격 확정 또는 노무/세무 자문 대체

## 6. 주요 사용자 흐름

### 6.1 데모 체험 흐름

1. 사용자가 로그인 화면에 진입한다.
2. "데모용 로그인으로 바로 체험하기"를 선택한다.
3. 퇴직 또는 이직 이벤트를 선택한다.
4. 진단 폼에 퇴사 사유, 다음 일자리 상태, 고용보험 기간, 소득 상태, 나이, 지역 등을 입력한다.
5. 서버 또는 데모 로직이 리포트 미리보기를 생성한다.
6. 사용자가 결제 화면에서 BASIC 또는 PLUS 플랜을 선택한다.
7. 데모 결제를 통해 결제 완료 상태로 전환한다.
8. 상세 리포트에서 우선순위, 예상 금액, 필요 서류, 공식 링크, 추가 공공혜택 후보를 확인한다.
9. PLUS 플랜이면 PDF 저장 화면과 AI 질문 기능을 사용할 수 있다.

### 6.2 실제 로그인 기반 흐름

1. 사용자가 카카오 또는 네이버 로그인 버튼을 선택한다.
2. 백엔드가 OAuth 제공자 인가 화면으로 리다이렉트한다.
3. 콜백에서 인가 코드를 토큰으로 교환하고 JWT를 발급한다.
4. 사용자는 약관 동의 후 온보딩/진단/리포트 흐름을 진행한다.

소셜 로그인은 환경변수에 카카오/네이버 클라이언트 정보가 설정되어 있어야 실제 제공자 연동으로 동작한다. 로컬/데모 환경에서는 mock OAuth 또는 브라우저 데모 세션을 사용할 수 있다.

## 7. 기능 요구사항

### 7.1 인증 및 약관

구현 상태: 구현됨, 환경 설정 의존

요구사항:

- 사용자는 카카오 또는 네이버 OAuth 로그인을 시작할 수 있어야 한다.
- OAuth 콜백 처리 후 access token과 refresh token을 발급받아야 한다.
- 사용자는 서비스 이용약관, 개인정보처리방침, 선택 마케팅 동의 상태를 저장할 수 있어야 한다.
- access token이 만료되거나 서버 재시작으로 인증이 무효화되면 로그인 화면으로 유도해야 한다.
- 데모 체험은 실제 OAuth를 거치지 않고 브라우저 로컬 세션으로 시작할 수 있어야 한다.

주의사항:

- 실제 소셜 로그인 성공 여부는 운영 환경변수와 각 제공자 개발자센터의 redirect URI 설정에 의존한다.
- 데모 세션은 실제 JWT가 아니며, 일부 기능은 프론트 데모 로직 또는 공유 데모 계정으로 처리된다.

### 7.2 온보딩 및 생애 이벤트 선택

구현 상태: 구현됨

요구사항:

- 사용자는 퇴직 또는 이직 이벤트를 선택할 수 있어야 한다.
- 결혼, 출산휴가, 육아휴직 등 준비 중 이벤트는 선택 불가 상태로 보여줄 수 있다.
- 선택한 이벤트는 진단 입력 화면으로 전달되어 질문 문구와 분석 기준에 반영되어야 한다.

현재 MVP 기준:

- 주요 시연 대상은 퇴직과 이직이다.
- 백엔드 enum에는 실직 유형도 존재하지만, 프론트의 핵심 진단 플로우는 퇴직/이직 중심으로 구성되어 있다.

### 7.3 진단 입력

구현 상태: 구현됨

요구사항:

- 사용자는 다음 정보를 입력할 수 있어야 한다.
  - 생애 이벤트 유형
  - 퇴직/마지막 근무일
  - 퇴사 또는 이직 사유
  - 다음 일자리 상태
  - 다음 일자리 시작일
  - 고용보험 가입 개월 수
  - 현재 소득 상태
  - 거주 지역
  - 나이
  - 근속연수
  - 가구 형태
  - 연소득 구간
  - 재산 구간
  - 주거 형태
  - 자녀/부양가족/기초생활수급/차상위/한부모/장애 여부

- 진단 제출 시 필수 항목이 비어 있으면 사용자에게 오류를 보여줘야 한다.
- 월 평균임금은 초기 진단에서 저장하지 않고, PDF 저장 시점에 별도 입력받을 수 있어야 한다.
- 기존 프로필 정보가 있으면 지역, 가구, 소득, 자산, 주거 정보를 진단 폼에 자동 채울 수 있어야 한다.

### 7.4 룰 엔진 기반 리포트 생성

구현 상태: 구현됨

요구사항:

- 사용자의 진단 정보를 바탕으로 서버 룰 엔진이 리포트를 생성해야 한다.
- 리포트는 절차별 신청 가능성, 우선순위, 사유, 기한, 필요 서류, 공식 링크를 포함해야 한다.
- 현재 리포트 항목은 다음 절차를 대상으로 한다.
  - 실업급여(구직급여)
  - 건강보험 임의계속가입
  - 국민연금 납부예외
  - 세금 정산 체크
  - 퇴직금 정산 확인

- 리포트 항목은 우선순위에 따라 정렬되어야 한다.
- 신청 가능성이 낮은 항목도 필요 시 "확인 필요" 또는 낮은 우선순위로 안내할 수 있어야 한다.

제한사항:

- 룰 엔진 결과는 행정기관의 최종 수급 판정이 아니다.
- 리포트는 사용자가 공식 사이트 또는 기관에서 확인해야 할 체크리스트를 제공한다.

### 7.5 예상 금액 계산

구현 상태: 구현됨

요구사항:

- 리포트는 입력값을 기반으로 예상 수령액 또는 월별 절감액을 계산해야 한다.
- 실업급여는 고용보험 가입기간, 나이, 월 평균임금 또는 법정 하한/상한액을 기준으로 예상 범위를 계산할 수 있어야 한다.
- 퇴직금은 근속연수와 월 평균임금 또는 연소득 구간을 기준으로 추정할 수 있어야 한다.
- 건강보험/국민연금 항목은 월 평균임금이 있을 때 예상 절감액 또는 부담액 안내를 제공할 수 있어야 한다.
- 금액은 "예상치"임을 명시해야 한다.

기준:

- 금액 산식은 서버의 `BenefitEstimationService`에 구현되어 있다.
- 정책 상수는 2026년 기준으로 작성되어 있으며, 실제 금액은 개인 상황과 기관 판단에 따라 달라질 수 있다.

### 7.6 결제 전 미리보기

구현 상태: 구현됨

요구사항:

- 사용자는 결제 전 리포트의 요약 제목, 요약 메시지, 발견 항목 수, 핵심 항목 일부를 볼 수 있어야 한다.
- 예상 수령액 범위를 계산할 수 있으면 미리보기 화면에 표시해야 한다.
- 나머지 상세 사유, 전체 필요 서류, 공식 링크는 결제 후 공개할 수 있다.
- 이미 결제된 리포트의 미리보기 URL에 접근하면 상세 리포트로 이동해야 한다.

### 7.7 결제 및 플랜

구현 상태: 구현됨, 토스는 테스트 결제

요구사항:

- 사용자는 BASIC 또는 PLUS 플랜을 선택할 수 있어야 한다.
- BASIC 가격은 6,900원, PLUS 가격은 13,900원으로 처리한다.
- 결제 전 신청 가능 또는 확인 필요 항목 수를 보여줘야 한다.
- 신청 가능 항목이 없으면 결제 진행을 제한해야 한다.
- 데모 결제는 외부 결제창 없이 결제 완료 상태로 전환할 수 있어야 한다.
- 토스 테스트 키가 설정된 경우 토스 테스트 결제창을 열고 승인 콜백을 처리할 수 있어야 한다.

플랜 차이:

- BASIC: 상세 리포트 열람
- PLUS: 상세 리포트, AI 질문 10회, PDF 저장 화면 이용

제한사항:

- 현재 결제 흐름은 데모 결제와 토스 테스트 결제를 중심으로 구현되어 있다.
- 운영 결제의 정산, 환불, 영수증, 결제 취소 관리는 MVP 범위 밖이다.

### 7.8 상세 리포트

구현 상태: 구현됨

요구사항:

- 결제 완료된 사용자만 상세 리포트를 볼 수 있어야 한다.
- 상세 리포트는 예상 총 수령액, 월별 절감액, 우선순위별 절차 목록을 보여줘야 한다.
- 가장 먼저 해야 할 일을 별도 영역으로 강조해야 한다.
- 각 절차는 필요 서류와 공식 신청 링크를 제공해야 한다.
- 결제하지 않은 사용자가 상세 리포트에 접근하면 미리보기 또는 결제 흐름으로 유도해야 한다.

### 7.9 공공혜택 DB 기반 추천

구현 상태: 구현됨, 환경 설정 의존

요구사항:

- 서버는 `gov24_benefit_cache` 테이블에서 공공혜택 후보를 읽어와야 한다.
- 런타임에서는 정부24 API를 매번 직접 호출하지 않고 DB 캐시를 조회해야 한다.
- 후보는 사용자의 이벤트 유형, 리포트 절차, 지역, 나이, 근속연수, 고용보험 기간, 퇴사 사유 등을 기준으로 필터링되어야 한다.
- 명확히 조건과 맞지 않는 후보는 제외해야 한다.
- 나이/근속연수 등 추가 입력이 필요하면 `pendingBenefits`와 `requiredForMatching`으로 분리해야 한다.
- 각 후보는 `relevanceScore`, `fitLevel`, `priorityGroup`, `matchedKeyword`, `sourceType`을 포함해야 한다.

검증된 동작:

- 공공혜택 기능을 켠 로컬 검증에서 결제 완료 리포트 조회 시 DB 출처의 추가 혜택 후보 6개가 반환되었다.
- OpenAI를 끈 상태에서도 서버 규칙 점수 기반 추천이 동작했다.

환경 조건:

- `GOV24_PUBLIC_SERVICE_ENABLED=true`일 때 상세 리포트 조회에 공공혜택 추천이 포함된다.
- 로컬 프로필은 H2 인메모리 DB와 `data.sql` 시드로 동작한다.
- Supabase/Postgres를 운영 DB로 사용하려면 Spring datasource JDBC URL, username, password가 운영 환경에 설정되어야 한다.

### 7.10 공공혜택 AI 재랭킹 및 요약

구현 상태: 구현됨, OpenAI 비용 발생 가능

요구사항:

- OpenAI가 활성화된 환경에서는 공공혜택 후보를 OpenAI Responses API로 전달해 우선순위와 사용자 친화적 요약을 보강할 수 있어야 한다.
- AI는 서버가 제공한 후보 JSON 안에서만 요약해야 하며, 새로운 금액이나 수급 조건을 만들어내면 안 된다.
- AI 응답은 구조화된 JSON schema로 받아야 한다.
- OpenAI 호출 실패 시 서버 규칙 기반 정렬 결과로 fallback되어야 한다.

중요한 제품 표현:

- DB 조회 자체와 서버 점수 계산은 OpenAI 비용이 들지 않는다.
- `OPENAI_ENABLED=true`이면 리포트 상세 조회 중 공공혜택 재랭킹/요약을 위해 OpenAI API가 호출될 수 있다.
- 따라서 PRD와 데모 설명에서는 "AI가 원자격을 판정한다"가 아니라 "서버 점수 결과를 AI가 재정렬하고 쉽게 설명한다"로 표현해야 한다.

검증된 동작:

- OpenAI 활성화 상태에서 리포트 상세 조회 시 공공혜택 6개 모두 `aiSummary`가 채워졌다.
- 같은 입력에서 AI가 단순 점수순과 다른 순서로 구직급여 관련 항목을 우선 배치했다.

### 7.11 리포트 기반 AI 질문

구현 상태: 구현됨, OpenAI 활성화 여부에 따라 실제 LLM 또는 서버 fallback

요구사항:

- PLUS 리포트 결제자는 리포트 기반 AI 질문 기능을 사용할 수 있어야 한다.
- 질문 가능 횟수는 10회로 제한한다.
- 사용자 질문과 AI 답변은 리포트별 메시지로 저장되어야 한다.
- 질문 1건마다 사용 횟수를 1회 차감해야 한다.
- BASIC 또는 미결제 리포트에서는 AI 질문을 제한해야 한다.
- OpenAI가 꺼진 환경에서는 리포트와 예상 금액을 근거로 한 결정적 fallback 답변을 제공해야 한다.

제한사항:

- AI 답변은 리포트 내용을 설명하는 보조 기능이며, 공식 행정 판단이나 법률 자문을 대체하지 않는다.
- OpenAI가 켜진 환경에서는 실제 API 비용이 발생할 수 있다.

### 7.12 PDF 저장

구현 상태: 구현됨

요구사항:

- PLUS 리포트 결제자는 PDF 저장 화면을 열 수 있어야 한다.
- PDF 저장 시 월 평균임금을 입력하거나, 입력하지 않고 저장하는 두 경로를 제공해야 한다.
- 월 평균임금을 입력하면 실업급여/퇴직금 등 금액을 더 구체적으로 계산해야 한다.
- 월 평균임금을 입력하지 않으면 범위와 산식 중심으로 리포트를 보여줘야 한다.
- PDF 저장은 브라우저 인쇄 기능을 통해 수행된다.

제한사항:

- 서버가 PDF 파일을 생성해 저장소에 업로드하는 방식은 아니다.
- 사용자가 브라우저의 인쇄/저장 기능으로 PDF를 저장한다.

### 7.13 필요 서류 모의 조회

구현 상태: 데모/모의 구현

요구사항:

- 상세 리포트는 각 절차별 필요 서류 목록을 보여줘야 한다.
- 사용자가 본인인증 모달을 완료하면 서류 조회 API를 호출할 수 있어야 한다.
- 서버는 필요 서류의 발급 기관 문자열을 기준으로 자동 조회 가능/직접 준비 필요 상태를 결정해야 한다.
- 자동 조회 가능으로 판정된 서류는 모의 다운로드 링크를 제공할 수 있다.

제한사항:

- 현재 본인인증은 실제 인증기관 연동이 아니라 프론트 모달 기반 데모 흐름이다.
- 현재 서류 조회는 실제 공공 마이데이터/전자문서지갑 연동이 아니라 mock 로직이다.
- PRD나 발표에서는 "모의 서류 조회" 또는 "향후 공공 마이데이터 연동 예정 구조"로 설명해야 한다.

### 7.14 커뮤니티

구현 상태: 구현됨

요구사항:

- 사용자는 퇴직/이직 카테고리의 커뮤니티 게시글을 볼 수 있어야 한다.
- 게시글 목록은 최신순 또는 인기순으로 볼 수 있어야 한다.
- 사용자는 게시글을 작성할 수 있어야 한다.
- 사용자는 게시글 상세를 보고 댓글을 작성할 수 있어야 한다.
- 사용자는 게시글 좋아요/좋아요 취소를 할 수 있어야 한다.
- 자신이 작성한 게시글과 댓글은 삭제할 수 있어야 한다.
- 데모 세션 또는 비로그인 요청은 공유 데모 계정으로 귀속될 수 있다.

제한사항:

- 커뮤니티는 퇴직/이직 중심이다.
- 결혼/육아휴직 등은 준비 중 카테고리로 표시될 수 있다.

## 9. 환경변수 및 기능 플래그

### 9.1 공공혜택 추천

- `GOV24_PUBLIC_SERVICE_ENABLED=true`이면 공공혜택 DB 캐시 추천을 사용한다.
- false이면 상세 리포트의 추가 공공혜택 목록은 비어 있을 수 있다.

### 9.2 OpenAI

- `OPENAI_ENABLED=true`와 `OPENAI_API_KEY`가 있으면 OpenAI 기반 기능이 사용된다.
- OpenAI는 리포트 AI 질문과 공공혜택 AI 재랭킹/요약에 사용될 수 있다.
- OpenAI를 끄면 리포트 AI 질문은 서버 fallback 답변으로 동작하고, 공공혜택 추천은 규칙 점수 기반으로 동작한다.

### 9.3 OAuth

- 카카오/네이버 client id, secret, redirect URI가 설정되어야 실제 소셜 로그인으로 동작한다.
- mock OAuth는 로컬/데모 개발용으로만 사용한다.

### 9.4 Toss

- `NEXT_PUBLIC_TOSS_CLIENT_KEY`가 `test_ck_`로 시작하면 프론트에서 토스 테스트 결제 버튼을 사용할 수 있다.
- 서버의 Toss secret key가 설정되어야 테스트 결제 승인 API가 정상 동작한다.

## 10. 수용 기준

### 10.1 기본 진단/리포트

- 사용자는 퇴직 또는 이직 이벤트를 선택할 수 있다.
- 필수 진단 입력을 완료하면 리포트 미리보기가 생성된다.
- 미리보기에는 발견 항목 수와 핵심 항목이 표시된다.
- 데모 결제 후 상세 리포트가 열린다.
- 상세 리포트에는 우선순위별 절차, 필요 서류, 공식 링크가 표시된다.

### 10.2 결제/플랜

- BASIC 결제 시 상세 리포트 열람이 가능하다.
- PLUS 결제 시 상세 리포트, AI 질문, PDF 저장 화면이 가능하다.
- 신청 가능 항목이 없는 경우 결제 진행을 제한한다.

### 10.3 공공혜택 추천

- `GOV24_PUBLIC_SERVICE_ENABLED=true`인 환경에서 리포트 상세 조회 시 DB 기반 공공혜택 후보가 반환된다.
- 후보는 점수, 적합도, 우선순위 그룹, 매칭 키워드를 포함한다.
- OpenAI가 꺼져도 규칙 점수 기반 추천이 동작한다.
- OpenAI가 켜져 있으면 `aiSummary`가 채워질 수 있다.

### 10.4 AI 질문

- PLUS 리포트에서만 AI 질문을 보낼 수 있다.
- 질문 1회 전송 시 사용 횟수가 증가한다.
- 10회 사용 후 추가 질문이 제한된다.
- OpenAI 비활성화 환경에서도 fallback 답변이 생성된다.

### 10.5 PDF/서류

- PLUS 리포트에서 PDF 저장 화면에 접근할 수 있다.
- 월 평균임금 입력/미입력 경로 모두 동작한다.
- 본인인증 모달 완료 후 모의 서류 조회 결과가 리포트에 반영된다.

### 10.6 커뮤니티

- 퇴직/이직 게시글 목록을 최신순/인기순으로 볼 수 있다.
- 게시글 작성, 상세 조회, 좋아요, 댓글 작성이 동작한다.
- 본인이 작성한 글과 댓글은 삭제할 수 있다.
